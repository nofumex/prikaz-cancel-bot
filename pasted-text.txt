# 1. Главная проблема: amoCRM не должна блокировать пользователя

Сейчас пользователь нажимает `Подготовить заявление`, а handler сначала делает `await crm.sync_case_event(...)`, и только потом отвечает пользователю. Если amoCRM отвечает медленно, бот молчит минуту.

Это недопустимо.

## 1.1. Главный принцип

Во всех пользовательских сценариях:

```text
сначала ответить пользователю
потом синхронизировать CRM в фоне
```

Нельзя:

```python
await crm.sync_case_event(...)
await message.answer(...)
```

Нужно:

```python
await message.answer(...)
schedule_crm_sync(...)
```

или:

```python
await message.answer(...)
asyncio.create_task(...)
```

но с правильной DB-сессией и обработкой ошибок.

## 1.2. Создать background CRM layer

Добавь модуль:

```text
app/services/crm_background.py
```

В нем реализуй:

```python
def schedule_crm_sync(
    settings: Settings,
    case_id: int,
    user_id: int,
    event_type: str,
    payload: dict | None = None,
) -> None:
    ...
```

и внутреннюю корутину:

```python
async def run_crm_sync_job(
    settings: Settings,
    case_id: int,
    user_id: int,
    event_type: str,
    payload: dict | None = None,
) -> None:
    ...
```

Требования:

* не передавать живой `AsyncSession` из handler’а в background task;
* внутри background task открывать новую DB session через `SessionLocal`;
* заново загружать `Case` и `User`;
* вызывать `crm.sync_case_event(...)`;
* ловить все исключения;
* писать ошибки в `crm_sync_logs`;
* не ломать Telegram/MAX flow;
* логировать длительность;
* не логировать токены.

Пример логов:

```text
CRM sync scheduled event=user_started_bot case_id=...
CRM sync start event=user_started_bot case_id=...
CRM sync done event=user_started_bot case_id=... duration_ms=...
CRM sync failed event=user_started_bot case_id=... duration_ms=... error=...
```

Если `CRM_SYNC_BACKGROUND=false`, можно выполнять sync синхронно только для debug scripts, но в production по умолчанию background.

## 1.4. Явное отключение CRM

Сейчас была проблема: CRM включается автоматически, если есть URL + token.

Нужно сделать так:

* если `AMOCRM_ENABLED=false` явно задано в `.env` — CRM выключена всегда;
* если `AMOCRM_ENABLED=true` — CRM включена;
* если `AMOCRM_ENABLED` не задан вообще — можно auto-enable при наличии `AMOCRM_BASE_URL` + `AMOCRM_ACCESS_TOKEN`.

То есть явный false имеет приоритет над auto-enable.

Проверить и исправить все места:

Добавить timing logs:

# 2. amoCRM: оставить ровно 5 этапов

Нужно оставить в воронке `Судебный приказ` ровно 5 рабочих этапов:

```text
1. Подписался на бота
2. Отправил приказ
3. Указал дату
4. Оплатил
5. Получил напоминание (не оплатил)
```

НЕ надо возвращать 11 этапов. НЕ надо менять на другие названия.

## 2.1. Mapping событий

Сделать `EVENT_STATUS_MAP`:

```python
EVENT_STATUS_MAP = {
    "user_started_bot": "Подписался на бота",

    "order_photo_uploaded": "Отправил приказ",

    "received_date_entered": "Указал дату",
    "envelope_photo_uploaded": "Указал дату",
    "ocr_completed": "Указал дату",
    "case_data_confirmed": "Указал дату",
    "preview_generated": "Указал дату",
    "payment_created": "Указал дату",
    "document_qa_failed": "Указал дату",
    "manager_requested": "Указал дату",

    "payment_paid": "Оплатил",
    "documents_delivered": "Оплатил",

    "reminder_sent": "Получил напоминание (не оплатил)",
    "payment_abandoned": "Получил напоминание (не оплатил)",
}
```

Важно:

* если человек получил reminder и потом оплатил — сделка должна перейти в `Оплатил`;

Технические события не должны создавать технические этапы.

Если в amoCRM уже есть старые этапы:

* не удалять автоматически;
* не ломать чужие этапы;
* создать/найти только нужные 5;

* в `scripts/check_amocrm.py` показать:

  * нужные этапы;

  * рекомендацию удалить лишние вручную, если нужно.

Чтобы CRM не тормозила:

* кэшировать pipeline_id;
* кэшировать status_id mapping;

* выполнять `ensure_statuses`:

  * при старте;
  * по admin-кнопке “Проверить CRM”;
  * в `scripts/check_amocrm.py`;
  * но не на каждом пользовательском событии.

# 3. Полноценный MAX-бот с тем же функционалом, что Telegram

Сейчас MAX-бот — только каркас. Нужно сделать MAX-бота полностью функциональным.

## 3.1. Изучить актуальную документацию MAX

Перед реализацией изучи официальную документацию MAX:

* общая документация API;
* `POST /messages`;
* `POST /subscriptions`;
* `GET /updates`;
* `POST /uploads`;
* объекты `Update`, `Message`, attachments, inline keyboard, callback.

Критично:

* использовать `platform-api2.max.ru`, а не устаревший/неправильный домен;
* токен передавать через заголовок `Authorization: <token>`;
* для production использовать webhook;
* long polling оставить только для dev/test;
* webhook должен проверять `X-Max-Bot-Api-Secret`, если задан secret;
* загрузка файлов идет через `POST /uploads` → upload URL → отправка message attachment.

## 3.3. MAX client

Переделать `app/adapters/max/client.py`.

Нужны методы:

```python
class MaxBotClient:
    async def get_me(self) -> dict: ...
    async def get_updates(self, marker: int | None = None, timeout: int = 30, limit: int = 100, types: list[str] | None = None) -> dict: ...
    async def send_message(self, chat_id=None, user_id=None, text="", keyboard=None, attachments=None) -> dict: ...
    async def answer_callback(self, callback_id: str, text: str | None = None) -> None: ...
    async def get_upload_url(self, upload_type: str) -> dict: ...
    async def upload_file(self, path: str | Path, upload_type: str) -> dict: ...
```

* для файлов использовать `type=file`;
* для изображений использовать `type=image`;
* не использовать `type=photo`.

# 11. Acceptance criteria

Проект можно считать готовым только если выполнено всё:

## CRM

* CRM не блокирует Telegram.
* CRM не блокирует MAX.
* Все CRM events идут background.
* `case:new` отвечает меньше 1 секунды при включенном AMOCRM_ACCESS_TOKEN.
* В amoCRM используется ровно 5 этапов:

  * `Подписался на бота`
  * `Отправил приказ`
  * `Указал дату`
  * `Оплатил`
  * `Получил напоминание (не оплатил)`
* Сделка реально двигается по этапам при действиях пользователя.
* Старые этапы не используются.
* Ошибка amoCRM не ломает пользовательский сценарий.
* `AMOCRM_ENABLED=false` реально выключает CRM.

## MAX

* MAX не каркас, а полноценный бот.
* MAX функционально равен Telegram.
* MAX умеет:

  * start/menu;
  * profile;
  * manager;
  * создать заявку;
  * принять фото/файл приказа;
  * принять конверт/дату;
  * показать карточку данных;
  * редактировать поля;
  * подтвердить данные;
  * сгенерировать preview PDF;
  * показать оплату;
  * проверить оплату;
  * получить full DOCX/PDF/instruction;
  * получить reminders;
  * синхронизироваться с CRM.
* MAX использует `platform-api2.max.ru`.
* MAX production использует webhook.
* MAX dev может использовать long polling.
* MAX webhook проверяет secret.
* MAX умеет upload/download файлов.
* MAX документы отправляются после оплаты.

## Документы

* Нет `№5`; есть `№ 5`.
* Нет `в городе Москва`; есть `г. Москва`.
* Нет `по договор №`; есть `по договору №`.
* Нет `коп..` и `руб..`.
* Бельский case генерирует 1 страницу.
* Нет странных пробелов.
* Суммы правильные.
* Preview PDF создается из full PDF.
* Платеж не создается без QA/PDF.

* `python -m py_compile` проходит.
* `pytest` проходит.
* `python scripts/smoke_test.py` проходит.
* `python scripts/check_amocrm.py` проходит.
